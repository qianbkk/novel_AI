一句话定义

「落笔」是一个用 多个 AI 协作 写长篇网文的工程。8 个写作 Agent 分工合作：先规划章节、逐章写作、做质量评分、差的就重写、写完归档。整条流水线跑完一本 50 章 ~14 万字的小说，端到端由 AI 接力完成。

它和"普通 AI 写小说"工具的根本区别

普通 AI 写小说

一个 prompt → 一章。上下文靠 prompt 塞进去，3 章之后就记不住"主角上次吃了什么药"了。没有质检，全靠运气。

落笔 · FirstDraft

8 个 Agent 各司其职；3 道记忆防线记住过去 200 万字；3 个 LLM 互相打分；差了就重写到 6.5 分以上；超预算就暂停。

它和"普通 AI 写小说"工具的根本区别（口语版）

把它想象成"8 个工种的剧组"：导演（planner）定大纲，编剧（writer）写初稿，校对（normalizer）改语病，3 个评委（checker）分别给情节/爽点/人物打分，差了就让编剧重写，剪辑（tracker）记下本章发生了什么，剧务（summarizer）把最近 5 章压缩成一张便条给下一任编剧。LangGraph 是"调度台"，按剧本顺序让这些角色挨个上场。

核心组件（8 个 Agent + 4 个支撑模块）

每一个 Agent 都是"一个活儿"，深度可调

为什么是 8 个而不是 1 个？

单一 LLM 在长篇任务里会"顾此失彼"。拆成 8 个角色，每个专注一件事：分而治之 + 互相打分。这是多智能体系统对抗"上下文漂移"的标准做法。

关键机制

这个项目最聪明的 4 个设计

① 5 维加权质检（防止"自欺欺人"）

一个 Checker 容易被 prompt 操纵，3 个独立 Checker 互相看不见打分。各打各的，然后加权： hook_power: 0.25 · shuang_density: 0.20 · character_voice: 0.15 · plot_logic: 0.25 · writing_naturalness: 0.15。 总分 < 6.5 自动触发 rewrite，最多重写 3 次（防无限循环），仍不过就推给 human_escalation。

② L2 热冷分离记忆（防止"爆预算"）

把章节状态拆成 4 层：hot（最近 5 章摘要 + 当前人物位置） · cold（世界事件 + 已关闭的剧情线） · constraints（伏笔 + 禁忌） · meta（最后更新章号）。 写作时只把 hot 喂进 prompt，cold 模糊查询。等 hot 超过 5 章自动冷压，永远不会有 prompt 爆掉的章节。

③ LangGraph SqliteSaver 断点续传（防止"前功尽弃"）

每一章完成后 LangGraph 自动把状态存进 backend/data/checkpoints.sqlite。进程崩了、网络断了、Ctrl-C 都能从最近一个 checkpoint 续上。这不是"重跑"，是精确到字段的"接着写"。

④ 桥接层（防止"老代码不能动"）

项目早期是从 novel_AI/ 那份命令式 Python 改造成的 FastAPI。新后端不能直接 import 老模块的 sys.path 注入（已经被改掉），所以加了一层 app/bridge/：只 import 已确认的 4 个接口（planner/init_arc/setting_sync/chapter_import），其余通过 output/setting_package.json 文件 + Subprocess 通信。
好处：老代码任何修改都不会"误伤"新后端；坏处：planner 完成后必须手动 pull-setting 把数据灌回 DB。
