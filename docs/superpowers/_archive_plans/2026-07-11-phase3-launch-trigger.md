# Phase 3 · 上线决策触发条件备忘

> 一页纸，给「未来的我」（或接手者）判断**何时从"原型阶段"切到"对外服务阶段"**。
> 不是代码、不是 roadmap，是触发器（trigger）的清单。
>
> 上线 = 正式启动 Phase 2（认证 / 迁移 / 反代），开启**多用户独立数据**模式。

---

## TL;DR

**保持原型的信号**（继续单租户本地使用）：

- 唯一访问者是你本人 / 1-2 个家人好友
- 数据只在本机 SQLite，存在损坏可接受
- 不需要 SaaS 化、不需要预留他人账号

**启动 Phase 2 的信号**（任一发生即触发）：

- 有非你本人的真实用户想用 + 你愿意让他/她用
- 数据完整性和隔离性变成真实责任（用户 A 的项目不能被 B 看到或破坏）
- 需要可观测性（审计日志 / 用量统计 / 备份）
- 上面的任意一条都不能靠"未来再补"糊弄过去——必须是现在要的

## 决策清单（打勾即上线）

满足**任意一项**就触发 Phase 2：

| # | 触发条件 | 验证信号 |
|---|---------|---------|
| 1 | **外部用户触达** | 有人给你 DM"我能不能也用来写小说"——不是客气话，是真的想注册 |
| 2 | **多设备共享** | 你想在手机 / 公司电脑 / 朋友家里都访问同一份数据 |
| 3 | **数据具备真实价值** | 你写过 10+ 章 + 章节被公开分享过，回退意味着公开数据消失 |
| 4 | **失败的成本变高** | DB 损坏 = 不是"重写一篇"，而是"客户 / 朋友的项目没了" |
| 5 | **3 个月内没重大 bug** | 自上次 Phase 2 / Phase 3 修复以来，3 个月 0 P0 / P1 issue |
| 6 | **决策疲劳** | 你发现自己每周都在想"这事什么时候收敛"——就该收敛了 |

不满足时：**不要因为"反正早晚要做"就提前动手**。这是 YAGNI。

## Phase 2 一旦启动，最低必备清单

不是今天的事，但触发器亮了的时候照这张表执行：

1. **认证层**
   - 用户表（email + 密码哈希 + bcrypt）
   - session-based 或 JWT（个人项目够用）
   - 每条 API 加 `Depends(get_current_user)`
   - 所有 `query.filter_by(project_id=...)` 改为 `query.filter_by(owner_id=current_user.id, project_id=...)`

2. **数据迁移**
   - 给 `Project.owner_id`（已预埋）回填你的账号 ID
   - 老用户数据原样保留

3. **引入 Alembic**
   - 替换裸 `ALTER TABLE + 裸 except` 模式
   - 每次 schema 变更走版本号迁移
   - 收益上线后才值得这个复杂度

4. **运行时收紧**
   - `NOVEL_PRODUCTION=1` 强制链路
   - `ALLOWED_ORIGINS` 只放真实前端域名
   - `ALLOWED_PROXIES` 配反代 IP
   - `RATE_LIMIT_EXEMPT_LOCALHOST=0`
   - 启动时 `MASTER_KEY` 已设置（fail-fast 已有，不变）

5. **备份策略**
   - 自动化每日快照（已有 manual dev.bat backup，半自动）
   - 上 OSS / Backblaze B2 之类异地（一次性预算几美元）

6. **审计日志**
   - 至少记：谁、在何时、对哪个项目、做了什么
   - 留存 30 天

## 不要做的事（提前做的反模式）

- ❌ **今天加 FastAPI Users / Auth0 / Clerk** —— 没用户，加了只是装饰，且这些库用一段时间就要换
- ❌ **今天上 Postgres** —— SQLite 够 1 个用户，迁移成本不该被预先支付
- ❌ **今天写 RBAC** —— 没有"共享权限"场景（已确认只"多用户各自独立数据"），RBAC 是为更复杂场景引入的过度抽象
- ❌ **今天搞 Sentry / Datadog** —— 1 个用户跑本地没有任何真实告警价值
- ❌ **今天上 Alembic** —— 见前述
- ❌ **今天做 OAuth** —— 不是外部用户使用 OAuth 的紧迫场景；本地 session/JWT 够用 5 年

这些事**触发后**做都来得及（删除成本远低于"过度工程然后删"）。

## 历史背景

2026-06-30 的三个发现（来自深度审查 + 独立审视）触发了这份备忘：

- **Finding #1**：无认证 + 直接 API 路由 ID 改写即可水平越权
- **Finding #2**：`audit_mode` 用 `os.environ`，多项目共用 backend 时会串扰（**Phase 3 已修**）
- **Finding #3**：`migrations.py` 的 `except Exception` 一刀切吞所有错误，启动不 fail-fast（**Phase 3 已修**）

加上 Phase 3 顺手埋的 `Project.owner_id` 列（nullable、不写不读，纯占位）。

Phase 2 才是真"上线"。
